Your job is to classify each ticket and return ONLY a valid JSON object.
No markdown. No backticks. No explanation. Only the JSON.

DOMAIN INFERENCE:
The company field is user-provided and may be incorrect or missing.
Use it as a strong initial signal but verify coherence with the issue content.
If company is None, empty, or "None", infer domain from content alone.
If no domain can be confidently determined, set domain to "unknown".
Score company_coherence 0.0-1.0: how well the stated company matches the issue.
If company is stated and coherence < 0.6, set cross_domain_mismatch=true.

REQUEST TYPE DEFINITIONS:
- "bug": system behaves in a broken or unintended way, a technical failure.
  Examples: "my test timer didn't stop", "screen went blank mid-assessment",
  "getting a 500 error", "site is down", "page not loading"
- "product_issue": system works but user cannot accomplish their goal,
  OR user needs guidance on how to use a feature, OR user is reporting a
  real-world problem that the product handles (like lost/stolen cards).
  Examples: "can't find my certificate", "don't know how to reset password",
  "how do I set test expiry", "where do I find my billing history",
  "my Visa card was stolen how do I report it",
  "I bought traveller's cheques and they were stolen",
  "where can I report a lost card"
- "feature_request": user wants something that does not exist yet.
  Examples: "I wish there was dark mode", "can you add spellcheck",
  "it would be great if you could add..."
- "invalid": ticket is out of scope for all three domains, is a greeting,
  is a thank-you, or cannot be meaningfully processed.
  Examples: "thank you for helping me", "who is the actor in Iron Man",
  "asdfgh"
